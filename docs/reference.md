Here is a complete architecture design report for your custom **Hybrid Orchestration Platform**. This design integrates the **Visual Usability of n8n**, the **Task Distribution of Conductor**, and the **Fault-Oblivious Replay of Temporal**, specifically architected to run on your **MongoDB** infrastructure.

# Architecture Design Report: The "Hybrid-Flow" Engine

## 1\. Executive Summary

The proposed platform, **"Hybrid-Flow,"** is a distributed, event-sourced orchestration engine. It separates the **Workflow Definition** (JSON DSL) from the **Execution Logic** (Go/Java Engine), utilizing **MongoDB** as the exclusive source of truth for both state and event history.  
**Key Architectural Goals:**

* **Traceability:** Implements "Time-Travel Debugging" via Event Sourcing (Temporal style) 1\.  
* **Usability:** A Visual Node Editor generates a JSON DSL (n8n style) 3\.  
* **Compatibility:** Runs natively on MongoDB via a custom Persistence Adapter 5\.  
* **Security:** "Zero-Knowledge" history via Client-Side Encryption (Temporal style) 6\.

## 2\. Core Concepts & Data Model

To support **Trace/Replay** on MongoDB, we cannot use simple task status updates (Conductor style). We must use **Event Sourcing**.

### 2.1 The Data Model (MongoDB Collections)

We will leverage MongoDB v4.0+ Multi-Document ACID Transactions.

* **workflow\_definitions**: Stores the JSON structure generated by the UI (n8n style).  
* **workflow\_executions**: The mutable state (Current Status, Start Time, Input).  
* **history\_events**: The **immutable** ledger.  
* **Schema:** { workflow\_id, run\_id, event\_id, event\_type, payload, timestamp }  
* **Shard Key:** workflow\_id  
* **Events:** WorkflowStarted, TaskScheduled, TaskCompleted, TimerStarted, TimerFired, SignalReceived.

### 2.2 The "Interpreter" Pattern

The Engine does not compile code. It acts as an **Interpreter** for the JSON definition.

* **User Action:** Drag-and-drop nodes in UI $\\rightarrow$ Generates workflow.json.  
* **Engine Action:** Loops through workflow.json. For each node, it checks history\_events.  
* *If event exists:* Restore variables from history (Replay).  
* *If event missing:* Execute the node (Action).

## 4\. Key Architectural Solutions to Your Constraints

### 4.1 Constraint: "Trace and Replay" (The Temporal Feature)

**Design:** The system will not update task status in place. Instead, every state change is an **Event**.

* **Trace:** To show the user *why* something failed, the UI queries the history\_events collection. It overlays these events onto the Visual Graph. Green nodes \= events found. Red nodes \= TaskFailed event found.  
* **Replay:** When a workflow crashes, the Worker restarts. It fetches the full history\_events list from MongoDB. It feeds these events into the Interpreter to fast-forward the state to the moment before the crash 1\.

### 4.2 Constraint: "MongoDB Only" (The Infrastructure)

**Design:** We will implement a custom Persistence Adapter.

* **Challenge:** Temporal/Cadence expects Row-Level Locking (SQL).  
* **Solution:** Use MongoDB FindOneAndUpdate with **Optimistic Locking**.  
* Every workflow execution has a next\_event\_id.  
* When writing a new event, the query must match: { \_id: workflow\_id, next\_event\_id: current\_id }.  
* If the match fails (another thread wrote), retry. This simulates the transaction isolation required by Cadence 5\.

### 4.3 Constraint: "5 Mins to 6 Hours" (Durable Timers)

**Design:** Do not use Thread.sleep().

1. **Mechanism:** When the JSON Interpreter hits a "Wait" node:  
2. Write TimerStarted event to MongoDB with fire\_time \= now \+ 6 hours.  
3. The Worker finishes its job and releases the thread (Resource usage \= 0).  
4. A background **Timer Component** (in the History Service) periodically polls MongoDB for fire\_time \<= now.  
5. When found, it pushes a TimerFired event to the workflow, waking up the Worker.

### 4.4 Constraint: "Dynamic Modification" (User Approval)

**Design:** Use the **Signal Pattern**.

1. **Workflow Logic:** The workflow includes a generic "Pause for Signal" node.  
2. **Process:**  
3. Workflow pauses at "Wait for Approval."  
4. User opens UI, modifies the *future* steps of the workflow JSON.  
5. User clicks "Approve."  
6. UI sends a **Signal** containing the *new* JSON structure.  
7. The Workflow receives the Signal, replaces its internal JSON definition variable with the new one, and resumes execution 7\.

### 4.5 Constraint: "Security" (Prevent Virtual Memory Leaks)

**Design:** **Data Converter (Codec) Pattern**.

* **Problem:** Sensitive data (e.g., PII) in local variables is written to history\_events.  
* **Solution:** The **Worker** (running on your secure server) holds a Symmetric Key.  
* *Input:* Worker receives data $\\rightarrow$ Decrypts.  
* *Persist:* Worker writes event $\\rightarrow$ Encrypts payload.  
* **MongoDB View:** The DB Admins only see encrypted binaries in the payload field.  
* **UI View:** The Debug UI requests the encrypted blob, passes it to a local Decryption Service (running on the developer's machine or secure zone), and renders the plain text 6\.

Since you are building this yourself using AI:

* **Phase 1: The Adapter (Hardest Part).**  
* *Prompt:* "Write a Go interface for a Workflow Persistence layer that supports 'CreateWorkflow', 'UpdateWorkflow', and 'AppendHistoryEvents'. Implement this interface using the MongoDB Go Driver with Multi-Document ACID Transactions."  
* **Phase 2: The Interpreter Engine.**  
* *Prompt:* "Create a simple Workflow Engine in Go that accepts a JSON DSL. It should iterate through steps. If a step is 'Activity', it should check a 'History' array. If found, skip. If not found, execute and append to History."  
* **Phase 3: The Visualizer.**  
* *Prompt:* "Create a React Flow component that renders a JSON workflow definition. Allow users to add nodes. On save, export the graph as a JSON object compatible with my Engine."
